# Morrow

> quietly in sync.

Morrow превращает сообщения из университетских чатов в проверяемые события. Он сохраняет источник каждого факта, отмечает сомнительные поля и помогает разобраться, почему дата или аудитория изменились.

## Демо

- Web: [morrow-sync.vercel.app](https://morrow-sync.vercel.app)
- Telegram: [@dekanat_panic_test_bot](https://t.me/dekanat_panic_test_bot)

Сайт устанавливается как отдельная программа: на Android/Windows/macOS используйте кнопку «Установить», на iPhone — Safari → «Поделиться» → «На экран Домой». Приложение запускается в отдельном окне и сохраняет local-first демо для базовой офлайн-работы.

Отправьте боту сообщение вроде «Лабу перенесли на 18.09 в 16:20, ауд. Б-304». Бот извлечёт событие, сохранит исходное сообщение как доказательство и вернёт карточку с кнопкой подтверждения и персональной ссылкой рабочей группы.

Фото, документ или голосовое тоже можно сохранить как источник: добавьте к вложению подпись с событием, датой, временем и аудиторией. Если подписи нет, бот объяснит, чего не хватает, вместо молчаливого игнорирования.

## Быстрый запуск

```bash
npm install
npm run dev
```

Для аккаунтов также нужны:

```text
NEON_AUTH_BASE_URL=https://...neonauth.../auth
NEON_AUTH_COOKIE_SECRET=случайная-строка-не-короче-32-символов
```

Создать cookie-secret можно командой `openssl rand -base64 32`. Он должен быть стабильным между деплоями и храниться только в защищённых environment variables.

Откройте [http://localhost:3000](http://localhost:3000), нажмите «Добавить» и вставьте сообщение:

```text
Лабу по программированию перенесли на 18.09 в 16:20, ауд. Б-304
```

## Проверки

```bash
npm run lint
npm run test
npm run build
# или всё сразу
npm run check
npm run test:e2e
```

Полный браузерный сценарий регистрации включается переменными `E2E_EMAIL_TEMPLATE` (например, disposable-адрес `morrow+{timestamp}@example.com`) и `E2E_PASSWORD`. Без них Playwright всё равно проверяет публичный демо-режим и безопасное поведение неподключённого Auth.

## Telegram

Не отправляйте токен бота в переписку и не добавляйте его в Git. Создайте локальный файл:

```bash
cp .env.example .env.local
```

Затем заполните новый `TELEGRAM_BOT_TOKEN` и публичный HTTPS-адрес приложения в `APP_URL`. `.env.local` уже исключён из Git. Команда ниже создаст случайный webhook-secret, сохранит его локально и зарегистрирует webhook:

```bash
npm run telegram:webhook
```

Webhook принимает только запросы с заголовком `X-Telegram-Bot-Api-Secret-Token`, извлекает событие, сохраняет его в PostgreSQL и отвечает пользователю интерактивной карточкой. Команда регистрации также публикует меню команд и включает обработку обычных сообщений и callback-кнопок.

## PostgreSQL

1. Создайте бесплатную базу Neon или другой PostgreSQL.
2. Добавьте строку подключения в `.env.local`:

```text
DATABASE_URL=postgresql://...
```

3. Примените миграции:

```bash
npm run db:migrate
```

Полезные команды:

```bash
npm run db:generate  # создать миграцию после изменения схемы
npm run db:studio    # открыть инспектор данных
```

Без `DATABASE_URL` приложение автоматически работает через версионированное локальное хранилище браузера.

Production использует управляемую Neon PostgreSQL. Сохранение нескольких связанных записей выполняется через batch API, совместимый с serverless HTTP-драйвером Neon.

## API

- `GET /api/health` — состояние базы и Telegram.
- `GET /api/session` — безопасный профиль текущей сессии.
- `GET|POST /api/workspaces` — пространства текущего пользователя.
- `POST /api/workspaces/claim` — первое безопасное подключение Telegram-группы.
- `POST /api/workspaces/invitations` — приглашение участника владельцем или администратором.
- `GET /api/events?groupId=...` — события демо или доступной группы.
- `POST /api/events` — сохранить событие после проверки membership.
- `PATCH /api/events/:id` — изменить статус события в пределах группы.
- `POST /api/telegram/webhook` — входящие Telegram updates.

## Архитектурное решение MVP

Первая версия намеренно использует детерминированный локальный парсер. Это позволяет отладить продуктовый сценарий и собрать набор примеров до подключения LLM. Пользовательские события сначала сохраняются в версионированную запись `localStorage`, а при подключении PostgreSQL объединяются с серверными данными. Следующий этап — очередь фоновых задач и гибридное извлечение: правила для очевидных полей, модель для сложного контекста, обязательное подтверждение человеком для низкой уверенности.
